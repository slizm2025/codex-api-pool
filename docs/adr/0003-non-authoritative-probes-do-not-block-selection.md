# Non-authoritative probes do not block Selection

Health Probes can use synthetic request shapes that some Upstreams reject even though real Codex traffic works, so probe failures are not all equally authoritative. The API Pool will only let Authoritative Probe Failures mark an Upstream unavailable for Selection; Non-representative Probe Results and inconclusive probe results remain selectable with lower priority until real Codex traffic or a Representative Model Probe confirms availability.

Representative success is time-bounded evidence, not a permanent whitelist. A real Codex success or Representative Model Probe success is fresh for 30 minutes by default; non-authoritative probe failures cannot override it while fresh, but Authoritative Probe Failures can. After freshness expires, the Upstream remains eligible at lower priority until new representative evidence refreshes it.

Representative evidence is scoped to `Upstream Key + protocol + model`, then summarized at the Upstream level for display. Selection should prefer keys with fresh representative evidence for the requested protocol and model instead of assuming one key's success proves all keys on the same Upstream.

Representative request templates are scoped to `protocol + client family`, not to an Upstream Key. A template captures Codex-native request shape, while each Representative Model Probe still injects the target Upstream Key at send time; Codex Desktop and Codex CLI templates may coexist when their request shapes differ.

Representative request templates may retain sensitive-but-necessary Codex context headers, such as attestation-like headers, only in memory and only until template freshness expires. They must never persist `authorization`, cookies, user input, arbitrary request `metadata`, or unmasked sensitive header values to Pool Configuration, Runtime State, logs, or the Management Dashboard.

Representative request template bodies may retain only structural request-shape fields needed to reproduce Codex-native traffic, such as `tools`, `tool_choice`, `parallel_tool_calls`, `store`, `reasoning`, `text`, and the minimal Responses API body shell. They must not retain original user `input`, prompt text, conversation history, file/image/audio payloads, `previous_response_id`, arbitrary `metadata`, or user/session/project identifiers. The template describes how Codex sends a request, not what the user asked.

Probe classification defaults to preserving Selection eligibility unless the response contains strong authoritative failure evidence. Probe outcomes are classified as `ok`, `authoritative_failure`, `non_representative`, or `inconclusive`; only `authoritative_failure` may block Selection. Merchant-specific or ambiguous `400`, `403`, `404`, and `5xx` responses are inconclusive unless accompanied by strong signals such as authentication/key failure, quota or rate-limit exhaustion, network/TLS/timeout failure, clearly unsupported model/API, or a target-protocol `2xx` response with an invalid output shape.

When a Management Dashboard Health Probe returns `non_representative` or `inconclusive` and no fresh Representative Request Template is available, the dashboard reports that the Upstream is not yet representative-verified instead of unavailable. The Upstream remains eligible for Selection at lower priority so real Codex traffic can capture representative evidence or produce an Authoritative Probe Failure. When a fresh template exists, the health action may run a Representative Model Probe; without one, it must not invent a more complex synthetic request and present it as Codex-native evidence.

Real Codex traffic uses the same evidence boundary but carries higher representative weight because the request shape is already native. A successful real Codex request refreshes representative success evidence for 30 minutes. An Authoritative Probe Failure from real Codex traffic may immediately enter the existing cooldown or unavailable logic. A non-authoritative real Codex failure records degradation evidence and may reduce priority or trigger a short operational cooldown, but it must remain distinguishable from a clear unavailable verdict.

Representative Request Templates are captured only from authenticated real Codex Model Interaction Requests at the model request interface, after Pool Token authorization and before forwarding to an Upstream. The captured artifact is a sanitized template, not the original request. Management API calls, Dashboard form input, model listing, synthetic Health Probes, and failed probe payloads cannot create representative templates. A failed real Codex request may still provide a request-shape template, but it does not provide success evidence.

Representative Request Templates are memory-only and disappear when their TTL expires or the API Pool process restarts. Representative success evidence and Authoritative Probe Failure evidence may be restored as Runtime State when they include timestamps, freshness windows, and cooldown expiry data. A restored representative success can influence Selection only while it remains fresh; a restored authoritative failure can influence Selection only according to its recorded cooldown or freshness window.

Representative evidence changes Selection probability rather than imposing a fixed priority order. Selection first builds the eligible candidate pool by excluding Disabled Upstreams, missing keys, active cooldowns, and active Authoritative Probe Failures. Within that pool, it continues to use weighted random Selection for load balancing. Fresh representative success may apply only a bounded positive multiplier; stale or unknown evidence should remain eligible so the pool can keep discovering usable Upstreams; non-authoritative degradation may apply a bounded negative multiplier but must not starve exploratory traffic.

Exploratory traffic is preserved through a minimum effective weight floor for candidates that are not authoritatively blocked. Unknown, stale verified, and non-authoritatively degraded candidates may receive less traffic than freshly verified candidates, but their effective weight must not be reduced to zero by evidence multipliers alone. Exploration never overrides user Disabled state, missing keys, active cooldowns, or clear model/protocol incompatibility.

Repeated non-authoritative failures from real Codex traffic may trigger a short operational cooldown, but they do not become Authoritative Probe Failures only by repetition. A single non-authoritative failure lowers confidence or weight without cooldown. By default, three consecutive non-authoritative real-traffic failures may trigger a short cooldown such as 30-120 seconds. A later representative success clears the consecutive non-authoritative failure count; any true Authoritative Probe Failure immediately follows the existing authoritative failure and cooldown rules.

Representative Model Probe success may set Health State to `ok` only when the state records its source and scope, such as `source = representative_probe`, `probeModel`, `keyLabel`, and a warning that it was verified by a representative Codex request template. Synthetic Health Probe success, Representative Model Probe success, and real Codex traffic success can all produce an `ok` Health State, but the Management Dashboard and Selection diagnostics must keep their sources distinguishable.

The initial Representative Request Template implementation is scoped to Codex Desktop because the current client traffic comes from Codex Desktop. Templates still carry a `client_family` field so future Codex CLI or other Codex client shapes cannot silently share a Desktop-derived template. If the client family cannot be identified reliably, it may be recorded as `unknown_codex_client` and used only within that family bucket.

Manual Management Dashboard health actions prefer Representative Model Probes when a fresh Codex Desktop Representative Request Template exists. If no fresh template exists, the action falls back to an ordinary Health Probe and must label the result as not representative of Codex Desktop availability. Scheduled background Health Probes do not replay Representative Request Templates in the initial implementation; this avoids repeated unattended reuse of sensitive context headers and keeps representative probing tied to explicit operator intent.

The API Pool captures a fresh Codex Desktop Representative Request Template whenever an authenticated real Codex Desktop Responses Model Interaction Request enters the model request interface. Capture happens after Pool Token authorization and before upstream forwarding, then sanitizes the request shape and replaces any previous in-memory template for that client family. Capture is independent from success evidence: a failed real Codex request may refresh the template shape, while only a successful model response with output refreshes representative success evidence.

Representative Model Probes choose their target model from the current Model Override first, then the most recently captured template model, then an explicit operator-selected model if such a control exists. Model listing must not be used to infer availability or choose a supposedly usable model for the probe; `/models` remains model discovery and display evidence only.

Representative Model Probe success requires a valid model response shape, not merely HTTP success. For Responses protocol probes, success requires HTTP `2xx` plus Responses output evidence such as `output_text`, a non-empty `output` array, or streaming `response.output_*` events, and must not be an explicit zero-output result. A target-protocol `2xx` response without valid output is an Authoritative Probe Failure for invalid output shape.

Representative Model Probes preserve the native request type captured in the Representative Request Template, including whether the original Codex Desktop request used streaming or non-streaming Responses. Probe execution may stop reading once it has enough response data to classify output validity, but it must not convert a native streaming template into a non-streaming synthetic request shape or vice versa.

Representative Model Probes replace user semantic content with minimal harmless probe content while preserving the captured native request structure. The probe may use a small input such as `Respond with OK.` and minimal probe instructions, while retaining structural fields from the template such as `tools`, `tool_choice`, `reasoning`, `text`, `store`, and `stream`. This preserves Codex Desktop request shape without retaining the user's original prompt or conversation history.

Probe classifiers use strong signal families rather than merchant-specific text as the basis for Authoritative Probe Failures. Strong signals include recognizable authentication or key rejection, quota or rate-limit exhaustion, network/TLS/socket/timeout failures, clearly unsupported model or endpoint, and target-protocol success status with invalid output shape. Ordinary merchant-specific `400`, `403`, `404`, `5xx`, HTML challenges, or vague invalid-request messages remain `inconclusive` or `non_representative` unless they contain a strong signal.

The meaning of an error depends on whether the request was representative. For example, `invalid_responses_request` or `invalid Codex request` from a synthetic Health Probe is a Non-representative Probe Result and must not block Selection. The same error from a Representative Model Probe or real Codex traffic is authoritative evidence for that represented request shape because the Upstream rejected the Codex Desktop-shaped request itself.

This trades some extra real-traffic attempts against false negatives for Codex-context-gated Upstreams. Diagnostics must make the uncertainty visible so an unknown or stale Upstream is not confused with a freshly verified healthy one.
