## Problem Statement

Codex sends Model Interaction Requests to the API Pool using the Responses API shape. Even when the user asks for ordinary text, those requests may carry Responses-only Features such as native tool declarations, reasoning items, multimodal content blocks, Responses state fields, and output-shaping options. When no Native Responses Route is available, otherwise healthy Chat Completions or Anthropic Messages Upstreams can be skipped or rejected with a 422 because the API Pool cannot safely prove the request is compatible.

The user needs the API Pool to support cross-protocol conversation in a controlled way: use a non-native adapter only when the user has enabled Adapter Compatibility Mode, convert every documented equivalent field, downgrade only when the target adapter has a weaker equivalent, strip only when no documented equivalent exists, and explicitly report what changed for that request.

## Solution

Add a complete Adapter Compatibility Mode for `/v1/responses` requests routed to Chat Completions or Anthropic Messages Upstreams. The mode should preserve the existing preference for a Native Responses Route. If no native candidate is available and the relevant adapter is enabled, the API Pool should build an adapter-specific compatibility plan, transform the request into the target protocol, forward it, and wrap the response back into a Responses-shaped result.

Every compatibility decision must be observable. The API Pool should report `converted`, `downgraded`, and `stripped` tool types, input item types, content types, output format types, and fields through response headers, Recent Request Timeline entries, and Management Dashboard surfaces. Silent deletion of user content or tool intent is not acceptable.

## User Stories

1. As a Codex desktop user, I want ordinary `你好`-style prompts to reach a healthy Chat Completions Upstream when no Native Responses Route exists, so that simple conversation does not fail because Codex included Responses-only metadata.
2. As a Codex desktop user, I want ordinary prompts to reach a healthy Anthropic Messages Upstream when I select a Claude model, so that cross-model conversation works through the same local API Pool.
3. As a Codex desktop user, I want the API Pool to prefer a Native Responses Route when one is available, so that full Responses semantics are preserved whenever possible.
4. As a Codex desktop user, I want Adapter Compatibility Mode to be opt-in, so that strict behavior remains available for workflows that require exact Responses behavior.
5. As a Codex desktop user, I want `input_text` content to be converted to target text content, so that my prompt text is preserved.
6. As a Codex desktop user, I want `input_image` data URLs and URLs to be converted to the target image content shape, so that image prompts still work through non-native Upstreams.
7. As a Codex desktop user, I want `input_image.file_id` to be converted when the target adapter has a documented file source, so that uploaded image references are not silently dropped.
8. As a Codex desktop user, I want `input_file.file_id` or `input_file.file_data` to be converted to Chat Completions file content parts, so that file prompts work on compatible Chat Upstreams.
9. As a Codex desktop user, I want `input_file.file_url` to be converted to an Anthropic document source, so that URL-based file prompts work with Claude-compatible Upstreams.
10. As a Codex desktop user, I want Chat-targeted `input_file.file_url` to be stripped with diagnostics when no documented Chat equivalent exists, so that I know exactly why the file was not forwarded.
11. As a Codex desktop user, I want Responses `function_call` and `function_call_output` history to be represented in Chat or Anthropic tool history, so that multi-turn tool conversations can continue.
12. As a Codex desktop user, I want Responses `custom_tool_call` and `custom_tool_call_output` history to be represented as far as the target adapter allows, so that Codex native tool flows do not fail unnecessarily.
13. As an operator, I want Responses `function` tools to become Chat function tools or Anthropic client tools, so that ordinary tool calling continues through non-native Upstreams.
14. As an operator, I want Responses `custom` tools to become Chat custom tools or be downgraded to Anthropic client tools, so that custom tool intent is preserved where possible.
15. As an operator, I want Responses `namespace` tools to be expanded into ordinary tool declarations with stable names, so that namespaced tool suites remain callable on adapters without namespace support.
16. As an operator, I want Responses web search tools to become Chat `web_search_options` or Anthropic server web search tools, so that hosted search intent is preserved where the target protocol has support.
17. As an operator, I want Responses `tool_search` to become an Anthropic tool search server tool when possible, so that Claude-compatible Upstreams can use the closest documented capability.
18. As an operator, I want Responses-only tools with no target equivalent to be stripped and reported, so that unsupported capability declarations do not block ordinary conversation.
19. As a Codex desktop user, I want `reasoning.effort` to map to the target effort field, so that model effort preferences are preserved when supported.
20. As a Codex desktop user, I want prior Responses reasoning input items to be stripped and reported, so that private or non-portable reasoning state is not forwarded into an incompatible protocol.
21. As a Codex desktop user, I want JSON output format requests to map to Chat `response_format` or Anthropic `output_config.format`, so that structured output requests continue to work.
22. As a Codex desktop user, I want unsupported output formats such as grammar to be stripped or rejected with clear diagnostics, so that I understand why exact output shaping was not preserved.
23. As an operator, I want state and output-shaping fields such as `previous_response_id`, `conversation`, `include`, `background`, and `truncation` to be stripped and reported, so that non-native adapters are not sent fields they cannot honor.
24. As an operator, I want response headers to show converted, downgraded, and stripped features, so that I can debug a single failed or degraded request from the client side.
25. As an operator, I want Recent Request Timeline entries to show compatibility decisions, so that I can diagnose cross-protocol behavior after the request completes.
26. As an operator, I want the Management Dashboard to expose the Adapter Compatibility Mode state, so that I can enable or disable the mode without editing configuration by hand.
27. As an operator, I want Chat and Anthropic adapter enablement to be independent, so that I can allow one target protocol without allowing the other.
28. As an operator, I want failed native attempts and compatibility fallback to preserve Retry, Fallback, Cooldown, Health State, Availability, and Usage behavior, so that compatibility does not distort pool operations.
29. As a developer, I want the response from Chat or Anthropic to be wrapped back into a Responses-shaped response, so that Codex receives the API shape it originally requested.
30. As a developer, I want streamed Chat or Anthropic responses to be adapted into Responses SSE events, so that streaming Codex UI behavior remains usable.
31. As a developer, I want tool-use responses to be represented as Responses output items, so that Codex can continue tool loops after a non-native adapter response.
32. As a developer, I want every adapter-specific conversion to be tested through the public API Pool request seam, so that regressions are caught at the behavior users actually depend on.

## Implementation Decisions

- Use the existing Adapter Compatibility Mode as the product boundary. The mode stays opt-in and only activates for `/v1/responses` Model Interaction Requests.
- Preserve Native Responses Route precedence. Compatibility conversion must not run before Selection has determined that no eligible native candidate is available.
- Resolve the target adapter from the Requested Model family after Model Override rules are applied. Claude-family requests target Anthropic Messages; other OpenAI-compatible requests target Chat Completions.
- Build a compatibility plan before mutating the outgoing body. The plan records the target adapter and the full diagnostics buckets for converted, downgraded, and stripped items.
- Treat content compatibility as adapter-specific. A content type may be convertible for one adapter and not for another.
- Convert text content blocks to the target text representation.
- Convert Responses image content to Chat image content parts or Anthropic image blocks. Preserve supported detail settings for Chat where documented.
- Convert Responses file content to Chat file content parts only when Chat has a documented `file_id` or `file_data` representation.
- Convert Responses file content to Anthropic document blocks when a documented PDF, text, content, or URL source can represent the file.
- Strip adapter-incompatible file source forms, such as Chat-targeted file URLs, with diagnostics instead of silently ignoring them.
- Convert Responses function tool declarations to the corresponding function/client tool shape.
- Convert Responses custom tool declarations to Chat custom tools. For Anthropic Messages, downgrade them to ordinary client tools with a string input schema.
- Expand namespace tools into ordinary tools with namespace-prefixed names.
- Convert or downgrade hosted web search according to the target adapter. Chat receives search options; Anthropic receives a server web search tool.
- Convert Anthropic-compatible tool search to the documented Anthropic tool search server tool. Strip Chat-targeted tool search with diagnostics.
- Convert tool choice where the target adapter can represent it. Downgrade unsupported Chat web search tool choice by removing it, and report the downgrade.
- Convert `reasoning.effort` to target effort controls. Strip the original Responses reasoning object after conversion because the full object is not portable.
- Strip prior reasoning input items because they represent Responses-specific state rather than user prompt content.
- Convert JSON output formats to target structured-output controls. Treat JSON object output for Anthropic as a downgrade to a generic object schema.
- Strip grammar output format when no target equivalent exists.
- Pass through Chat-specific request fields already documented by Chat Completions when targeting Chat.
- Map Anthropic-compatible top-level fields explicitly instead of passing arbitrary fields through.
- Wrap non-streaming Chat and Anthropic responses back into Responses-shaped JSON.
- Adapt streaming Chat and Anthropic events into Responses SSE events while preserving text deltas, completion events, usage where available, and tool call deltas where available.
- Report compatibility diagnostics through response headers, Recent Request Timeline entries, and dashboard presentation.

## Testing Decisions

- Tests should verify external behavior through the API Pool request seam rather than internal helper behavior. A good test sends a `/v1/responses` request to a configured pool, observes the fake Upstream payload, and asserts the client-facing response and compatibility diagnostics.
- Existing smoke tests are the primary testing seam because they already cover Selection, Retry, Fallback, protocol adapters, Usage, recent requests, Health State, and dashboard behavior.
- Add fake Chat Completions Upstream assertions for image content, file content, function tools, custom tools, web search downgrade, incompatible file URL stripping, and response wrapping.
- Add fake Anthropic Messages Upstream assertions for image blocks, document blocks, function/client tools, web search, tool search, reasoning effort mapping, structured output mapping, and response wrapping.
- Add compatibility-mode tests that confirm no native Responses attempt is made when no native candidate exists and the adapter mode is allowed.
- Add strict-mode tests that confirm native-only features still require a Native Responses Route when compatibility mode is disabled.
- Add diagnostics tests for each bucket: converted tool types, downgraded tool types, stripped tool types, converted input item types, stripped input item types, converted content types, stripped content types, converted output formats, downgraded output formats, stripped output formats, and stripped fields.
- Add tests for Recent Request Timeline compatibility metadata, not only response headers.
- Add tests for both non-streaming JSON responses and streaming SSE responses.
- Avoid tests that assert private helper function names or exact implementation structure.

## Out of Scope

- Building a perfect semantic clone of the Responses API on top of Chat Completions or Anthropic Messages.
- Supporting Responses stateful conversation semantics such as `previous_response_id` or `conversation` on non-native adapters.
- Implementing hosted tools with no documented target equivalent, such as Responses image generation on Chat Completions or Anthropic Messages.
- Uploading local files to target provider file APIs during compatibility conversion.
- Guaranteeing that tool call output shapes are lossless across protocols when the target protocol lacks an equivalent concept.
- Changing Selection, Retry, Cooldown, Health Probe, Availability, Billing, or Usage domain rules beyond what is required to route compatible requests.
- Enabling Adapter Compatibility Mode by default.

## Further Notes

- The domain glossary should continue to use `Chat Completions adapter` and `Anthropic Messages adapter` rather than model-family shorthand such as `Claude adapter`.
- Compatibility Stripping is acceptable only after checking documented target adapter fields. It must never be a silent cleanup step.
- Subfield losses should be treated as downgrades. For example, if a content block is preserved but a detail or source-specific option cannot be represented, the request diagnostics should show the lost subfield under downgraded fields.
- The PRD intentionally describes desired external behavior, not specific file paths or helper names, so future implementation agents can choose the smallest safe changes in the codebase.
