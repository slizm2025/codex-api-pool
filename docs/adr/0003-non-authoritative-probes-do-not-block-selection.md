# Non-authoritative probes do not block Selection

Health Probes can use synthetic request shapes that some Upstreams reject even though real Codex traffic works, so probe failures are not all equally authoritative. The API Pool will only let Authoritative Probe Failures mark an Upstream unavailable for Selection; Non-representative Probe Results and inconclusive probe results remain selectable with lower priority until real Codex traffic confirms availability.

Representative success is time-bounded evidence, not a permanent whitelist. A real Codex success is fresh for 30 minutes by default; non-authoritative probe failures cannot override it while fresh, but Authoritative Probe Failures can. After freshness expires, the Upstream remains eligible at lower priority until new representative evidence refreshes it.

Representative evidence is scoped to `Upstream Key + protocol + model`, then summarized at the Upstream level for display. Selection should prefer keys with fresh representative evidence for the requested protocol and model instead of assuming one key's success proves all keys on the same Upstream.

Representative Request Template metadata is scoped to `protocol + client family`, not to an Upstream Key. Template metadata captures freshness, model, and redacted replay-risk field names for diagnostics only. It is not a replayable request, and it is not used to synthesize Management API Health Probes.

Representative Request Templates may retain sensitive-but-necessary Codex context header names, such as attestation-like header names, only as redacted metadata and only until freshness expires. They must never persist `authorization`, cookies, user input, arbitrary request `metadata`, request bodies, or unmasked sensitive header values to Pool Configuration, Runtime State snapshots, logs, or the Management Dashboard.

Representative request template bodies are not retained. One-time fields such as attestation, nonce, turn state, signatures, or session-bound metadata make replay-based Management API testing unreliable, so the API Pool does not replay in-memory templates for availability checks.

Probe classification defaults to preserving Selection eligibility unless the response contains strong authoritative failure evidence. Probe outcomes are classified as `ok`, `authoritative_failure`, `non_representative`, or `inconclusive`; only `authoritative_failure` may block Selection. Merchant-specific or ambiguous `400`, `403`, `404`, and `5xx` responses are inconclusive unless accompanied by strong signals such as authentication/key failure, quota or rate-limit exhaustion, network/TLS/timeout failure, clearly unsupported model/API, or a target-protocol `2xx` response with an invalid output shape.

Manual Management Dashboard health actions run ordinary synthetic Health Probes first. When a Health Probe returns `non_representative` or `inconclusive`, the dashboard reports that the Upstream is not yet representative-verified instead of unavailable, includes the full captured Upstream probe result for diagnosis, and leaves the Upstream eligible for Selection unless a deterministic authoritative failure occurred.

The second validation layer is real Codex traffic through the normal model request interface. A successful real Codex request refreshes representative success evidence for 30 minutes. An Authoritative Probe Failure from real Codex traffic may immediately enter the existing cooldown or unavailable logic. A non-authoritative real Codex failure records degradation evidence and may reduce priority or trigger a short operational cooldown, but it must remain distinguishable from a clear unavailable verdict.

Representative Request Template metadata is captured only from authenticated real Codex Model Interaction Requests at the model request interface, after Pool Token authorization and before forwarding to an Upstream. Management API calls, Dashboard form input, model listing, synthetic Health Probes, and failed probe payloads cannot create representative template metadata. A failed real Codex request may still refresh metadata, but it does not provide success evidence.

Representative evidence changes Selection probability rather than imposing a fixed priority order. Selection first builds the eligible candidate pool by excluding Disabled Upstreams, missing keys, active cooldowns, and active Authoritative Probe Failures. Within that pool, it continues to use weighted random Selection for load balancing. Fresh representative success may apply only a bounded positive multiplier; stale or unknown evidence should remain eligible so the pool can keep discovering usable Upstreams.

Exploratory traffic is preserved through a minimum effective weight floor for candidates that are not authoritatively blocked. Unknown, stale verified, and non-authoritatively degraded candidates may receive less traffic than freshly verified candidates, but their effective weight must not be reduced to zero by evidence multipliers alone. Exploration never overrides user Disabled state, missing keys, active cooldowns, or clear model/protocol incompatibility.

Repeated non-authoritative failures from real Codex traffic may trigger a short operational cooldown, but they do not become Authoritative Probe Failures only by repetition. A single non-authoritative failure lowers confidence or weight without cooldown. By default, three consecutive non-authoritative real-traffic failures may trigger a short cooldown such as 30-120 seconds. A later representative success clears the consecutive non-authoritative failure count; any true Authoritative Probe Failure immediately follows the existing authoritative failure and cooldown rules.

Synthetic Health Probe success and real Codex traffic success can both produce an `ok` Health State, but the Management Dashboard and Selection diagnostics must keep their sources distinguishable. Synthetic success proves only the synthetic request shape; real Codex success refreshes representative evidence.

Successful probe or real traffic evidence can also learn an Upstream's Request Interface. Once the learned interface is Chat Completions, future Responses-shaped Codex traffic should prefer the Chat Completions adapter for that Upstream rather than repeatedly trying Native Responses first. This is a protocol routing preference, not an Authoritative Probe Failure for Responses.

Real successful model traffic can additionally learn a Forwarding Strategy scoped to `Upstream + Requested Model`. Later requests for the same pair should use that strategy directly, such as Native Responses, Chat Completions, or Chat Completions with Adapter Compatibility Mode, until a later successful strategy supersedes it.

Synthetic Health Probes choose their target model from the current Model Override. Model listing must not be used to infer availability or choose a supposedly usable model for the probe; `/models` remains model discovery and display evidence only.

Synthetic Health Probe success requires a valid model response shape, not merely HTTP success. For Responses protocol probes, success requires HTTP `2xx` plus Responses output evidence such as `output_text`, a non-empty `output` array, or streaming `response.output_*` events, and must not be an explicit zero-output result. A target-protocol `2xx` response without valid output is an Authoritative Probe Failure for invalid output shape for that synthetic shape.

Probe classifiers use strong signal families rather than merchant-specific text as the basis for Authoritative Probe Failures. Strong signals include recognizable authentication or key rejection, quota or rate-limit exhaustion, network/TLS/socket/timeout failures, clearly unsupported model or endpoint, and target-protocol success status with invalid output shape. Ordinary merchant-specific `400`, `403`, `404`, `5xx`, HTML challenges, or vague invalid-request messages remain `inconclusive` or `non_representative` unless they contain a strong signal.

The meaning of an error depends on whether the request was representative. For example, `invalid_responses_request` or `invalid Codex request` from a synthetic Health Probe is a Non-representative Probe Result and must not block Selection. The same error from real Codex traffic is authoritative evidence for that represented request shape because the Upstream rejected the Codex Desktop-shaped request itself.

This trades some extra real-traffic attempts against false negatives for Codex-context-gated Upstreams. Diagnostics must make the uncertainty visible so an unknown or stale Upstream is not confused with a freshly verified healthy one.
